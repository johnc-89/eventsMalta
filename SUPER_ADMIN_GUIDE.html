Admin cheat sheet

Everything an admin or super-admin needs to know to run Events Malta day to day — and what each button actually does. Rows marked super only apply only to super-admins.

Production: eventsmalta.org · Stack: Next.js 14 · Supabase · Tailwind · Vercel · Last updated to match the CRM build (May 2026)

TL;DR — your daily five

  1. Approve / reject submitted events → /admin
  2. Add a new outreach lead → /admin/crm/leads → + Add Lead
  3. Move a lead through the pipeline → click its status pill in the table
  4. Promote / demote / suspend a user → /admin/users
  5. See where you stand → /admin/crm/dashboard for the funnel, /admin for the queue

How the site works (mental model)

Public visitors browse /events and view individual events at /events/[slug]. No login needed.

Signed-up users can post events at /events/create. Posts go to pending_review by default.

Trusted uploaders are users you've blessed — their posts auto-approve, no review queue.

Admins see /admin with the queue and can approve / reject with a reason.

Super-admins can do everything admins can, plus suspend or hard-delete users, and access the CRM for outreach.

The whole site speaks to a single Supabase project. Authentication, the database, file storage, and realtime updates all run there. Vercel hosts the Next.js front end and auto-deploys when main changes on GitHub.

One-line summary. Admin is a moderator: events, tags, and users-other-than-admins. Super admin does everything admin does, plus owns the brand / CRM / site-customisation surface and is the only role that can suspend or delete staff and live content.

Roles live in profiles.role. Promote yourself with UPDATE profiles SET role='super_admin' WHERE id='your-uuid'; in the Supabase SQL editor — but only do this for accounts you own.

Managing users — /admin/users

  • Bulk invite: paste comma/newline-separated emails, each gets a branded magic-link invite to set their password. Already-existing emails are flagged "already a user" — no duplicate accounts.
  • Search: filter by name or email instantly.
  • Grant Auto-Upload (toggle): turns a user into a trusted uploader — their events skip the review queue.
  • Promote to Admin (toggle): grants moderation powers. Demote returns them to user.
  • Suspend super only: account becomes read-only. They see a banner explaining it; they can browse but not create or save anything. Reversible via Unsuspend.
  • Delete super only · destructive: hard-deletes the auth account and soft-deletes their events. They cannot log back in. Cannot be undone.
Suspend before delete. Suspending is reversible and gives you a beat to confirm the user is actually behaving badly. Delete is final.

Event review pipeline — /admin

An event goes through these statuses:

draft → pending_review → approved · or rejected · or organiser sets cancelled

  • Approve: event becomes public on /events and the organiser gets an email.
  • Reject: must include a reason (visible to the organiser). They get an email with the reason and can edit + resubmit.
  • Trusted uploaders skip pending_review entirely.

Tags & categories — /admin/tags

Categories are the top-level taxonomy (Party, Comedy, Theatre, etc.) — visible on the homepage cards. Tags are free-form labels organisers attach to their events. Edit the canonical list here so the dropdown stays clean.

What suspended users see

A persistent banner at the top of every page. They can still browse and view, but can't:

  • Create or edit events
  • Save events
  • Edit profile fields

Unsuspending restores everything in one click.

Event imports — /admin/sources super only

The site can pull events from other Malta event websites so visitors find everything in one place. Imports are owned by a dedicated Events Malta profile (the “aggregator user”), always land in pending_review, and show a source-attribution line on the public card.

Status today. The data model, settings and source list are live. The scrape pipeline itself (Phase 2) is not yet built — every source ships disabled and Run now is greyed out until the matching adapter is written. Once an adapter ships, you can flip the source on and Vercel Cron starts importing on the schedule below.

Sources — /admin/sources

Lists every external site we pull from. Each row has:

  • Enable toggle — master switch. While off, the source is invisible to the importer and to Vercel Cron.
  • Run now — fires an import immediately (admin-triggered). Disabled until the adapter exists.
  • Notes — internal-only field. Use it for adapter caveats, partnership status, robots.txt quirks.
  • Recent runs — last 10 import runs with insert / update / skip / excluded / error counts. excluded = blocked by the political-content filter.

One-time setup. The first time you visit the page you'll see an amber banner asking you to create the aggregator user. Click it once. This creates an auth.users row with no usable password, a profiles row with role trusted_uploader and display name Events Malta, and stamps its UUID into site_settings.published.importers.aggregator_user_id. Sources can't be enabled until this is done.

Adding a new source requires a code change (each site needs its own adapter file at lib/importers/adapters/<name>.ts). The 8 announced Malta sources are already seeded — start by enabling the ones whose adapter has shipped.

Attribution & political filter — /admin/site/importers

Lives inside the Site Editor (so changes go through the normal draft → Publish flow).

  • Attribution toggle. When on, imported events show a small line on their card (default: “Imported from Festivals Malta”). The template uses {source} as a placeholder for the source's display name. Off makes imports indistinguishable from user submissions.
  • Hard-block keywords. Case-insensitive substring match against title + description + venue + organiser. A match means the event is never imported — counted as excluded in the run log. Pad bare initials with spaces ( pl , not pl) to avoid matching inside other words.
  • Soft-flag keywords. A match still imports the event, but it lands in pending_review with a visible flag for the moderator. Use this for ambiguous words like minister, parliament — a Heritage Malta talk by a minister isn't political, but it deserves a second look.
Hard rule. The site does not host party-political event listings. Keep the hard-block list current with active Maltese parties, candidates, and the words rally / campaign / partisan / manifesto launch. If you spot a political event that slipped through, add the missing phrase to the hard list and click Publish.

CRM — /admin/crm

Tracks outreach to potential event organisers from cold lead → signed-up user. Three pages, locked to super-admins.

The point of the CRM: work each lead through statuses (Not Contacted → Contacted → Responded → Converted). When a lead signs up using the same display name as the lead, the CRM auto-links them.

Dashboard — /admin/crm/dashboard

  • Six metric cards: Total leads · Not Contacted · Contacted · Responded · Converted · High Quality (with % of total).
  • Four bar charts: By status · By category · Quality distribution · Contact platform mix.
  • Numbers are live — open this in one tab while editing in another and watch them tick.

Leads page — /admin/crm/leads

Searching & filtering

  • Search box: matches across name, category, subtype, channel, email, phone, pitch, notes, and every URL field at once.
  • Filter dropdowns (Status / Quality / Category): each has its own internal search box for fast scanning of long lists.
  • Sort: click any column header. Click again to reverse.
  • Clear filters: appears in the top right when any filter is active.

Editing

  • Click any text cell → it becomes an input. Enter saves, Esc cancels. Saves are instant.
  • Status pill: dropdown — Not Contacted → Contacted → Responded → Converted → Rejected. Changing status off "Not Contacted" auto-stamps today's date in Last Interaction if it's blank.
  • Quality pill: dropdown — High / Medium / Low.
  • Email cell: renders as a clickable mailto: link.
  • Link cell: opens in a new tab.
  • Edit button (or click + Add Lead): opens the full modal with every field grouped — Identity, Contact, Notes, Dates — and a History footer.
  • Delete button: confirms first; cannot be undone (history is preserved, but the row is gone).

Sync indicator

Top-left of the CRM topbar:

  • Green dot · live: connected to Supabase realtime; remote changes appear instantly.
  • Amber pulsing · saving…: a write is in flight.
  • Grey · offline: connection lost; refresh once you're back online.

Import / Export — /admin/crm/import-export

  • Download CSV: every lead, every column. Open in Excel / Google Sheets.
  • Upload CSV: merges by name. Existing leads get updated; new ones get inserted. No duplicates.
  • Paste from spreadsheet or AI: paste tab-separated rows from Sheets/Excel or comma-separated rows from an AI. Header row auto-detected. Misaligned columns are content-classified (URLs go to website fields, emails to email, phones to phone, etc.).
  • Seed default leads: loads the original 185-row Malta outreach dataset. Idempotent — run it as many times as you like.
  • Database info: shows the connected Supabase project + total lead count.

Audit trail & history

Every field change writes a row to lead_history automatically (database trigger — cannot be bypassed by the front end). The Edit modal shows the last 5 changes for that lead at the bottom, including who, when, what changed, and old → new values.

For a full audit, query the table directly — see Useful SQL.

Lead → user conversion

The CRM links a leads.name to a profiles.display_name when status hits Converted. Two cases:

  1. User already signed up before status change → link is set immediately on the row (converted_user_id).
  2. User signs up later → a trigger on profiles watches for new rows; if the display_name matches a Converted lead, it backfills the link automatically.

Match is case-insensitive. If a lead's name doesn't exactly match what they typed at signup, you can fix it by editing the lead's name to match — the trigger picks it up on next save.

Deploys

  1. Code merges to main on GitHub.
  2. Vercel detects, builds, deploys. Takes 1–2 min.
  3. Production swaps over instantly when the build is green.

If a deploy fails, the previous build keeps serving — production never goes down from a bad commit. Check the Vercel dashboard for build logs.

Useful SQL (paste into Supabase SQL Editor)

List all super-admins 
select u.email, p.display_name, p.created_at
from profiles p
join auth.users u on u.id = p.id
where p.role = 'super_admin';
Promote a user to super-admin 
update profiles
set role = 'super_admin'
where id = (select id from auth.users where email = 'you@example.com');
Funnel snapshot — leads by status 
select status, count(*) from leads group by status order by count(*) desc;
Recent CRM activity (last 50 changes) 
select changed_at, changed_by, lead_name, field_name, old_value, new_value
from lead_history
order by changed_at desc
limit 50;
Leads converted to users 
select l.name, l.status, p.display_name, p.created_at as user_signed_up
from leads l
left join profiles p on p.id = l.converted_user_id
where l.status = 'Converted'
order by l.updated_at desc;
Pending events older than 7 days 
select id, title, organizer_id, created_at
from events
where status = 'pending_review'
  and deleted_at is null
  and created_at < now() - interval '7 days'
order by created_at;

Troubleshooting

I'm a super_admin but the CRM link doesn't show in the navbar

Refresh the page (the role is read once at login). If still missing, check profiles.role in Supabase — it must be exactly super_admin (not superadmin or super-admin).

CRM shows "no leads" even after seeding

Open the browser console — if you see "relation leads does not exist", the SQL migration supabase/migrations/0001_crm.sql hasn't been applied. Paste it into Supabase SQL Editor and run.

Sync indicator stuck on amber / grey

Realtime is having trouble. Try a hard refresh (⌘ Shift R / Ctrl Shift R). If it persists, check the Supabase realtime logs — typically a network blip. Edits still save; only the live updates from other users are affected.

An organiser says their event was rejected unfairly

The rejection reason is stored in events.rejection_reason and emailed to them. If you re-approve, run update events set status='approved', rejection_reason=null where id=… in SQL — or set status back via the admin dashboard.

I deleted a lead by mistake

The row is gone but its history is preserved. Check lead_history for the lead's last known state and recreate it with + Add Lead.

Vercel deploy failed

Check the Vercel dashboard's build log. Most common causes: a missing env var (NEXT_PUBLIC_SUPABASE_URL, NEXT_PUBLIC_SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY) or a TypeScript error. Production keeps serving the previous build until the next one is green.

Things to never do

  • Don't share the service-role key. It bypasses RLS. Lives only in Vercel env vars.
  • Don't delete users without suspending first. Delete is irreversible.
  • Don't edit production data via Supabase Table Editor when triggers exist (audit, last_interaction stamping). Use the UI or RPCs so triggers fire correctly.
  • Don't disable RLS. Even temporarily. The whole permission model rests on it.
  • Don't push to main without a passing build locally (npm run build) — it'll deploy the breakage.
